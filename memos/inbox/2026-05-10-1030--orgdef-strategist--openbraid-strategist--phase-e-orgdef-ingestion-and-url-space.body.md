# Phase E: orgdef ingestion + canonical position URL space

## Context

Companion to the 10:00 memo (`canonical-oagp-position-addressing-decided`) which established the architectural decisions. That memo detailed WHAT (URL shape, boot payload, mirror-list, etc.) but missed making one principle explicit. This memo corrects that omission AND scopes Phase E concretely.

## The missing principle (explicit, this time)

**Openbraid is the HOSTING layer; orgdef.openthing artifacts are the CONTENT layer.** Openbraid does NOT have its own data model for positions, relationships, governance, or recommended_patterns. It INGESTS catdef-substrate orgdef.openthing artifacts as the source of truth, stores them in their canonical JSON form, exposes them via the MCP URL space, and provides full-fidelity export to round-trip the same artifact back out.

This inversion matters for several non-obvious reasons:

1. **Mirror-list compatibility.** When an org declares both git and openbraid in `x.org.org_location` with one marked `authoritative: true`, mirroring is a file copy — not lossy translation. orgdef-derived-from-openbraid-data-model would require translation; orgdef-as-source-of-truth-stored-as-JSON is byte-equivalent.
2. **Full-fidelity export as load-bearing escape valve.** An org publishing only to openbraid can export at any time. The exported artifact IS the canonical artifact; no information is lost going from openbraid to file to git. This is what prevents openbraid-as-hosting from becoming openbraid-as-lock-in.
3. **Cross-protocol equivalence.** A position at `mcp.openbraid.app/scott/thingalog/strategist` returns content derived from the SAME orgdef.openthing artifact that `https://github.com/scott/thingalog/blob/master/org/thingalog-organization.openthing` resolves to. Same content; different access; protocol-as-mechanism-only.
4. **No invented identity model.** Positions are identified by their `id` field in the orgdef artifact. Relationships are typed edges declared in the artifact. Governance is the `governance_model` field. Recommended patterns are the `recommended_patterns.general` array. Openbraid does not invent parallel identifiers, parallel relationship types, or parallel governance constructs; everything traces back to the orgdef artifact.

If openbraid Phases A–D made any decisions that violate this principle (e.g., positions stored as openbraid-native rows with orgdef.openthing being an export-only afterthought; relationships modeled in openbraid's own schema rather than read from the artifact's `relationships` array), Phase E surfaces that and refactors. If Phases A–D got this right by inference from prior conversation, Phase E just codifies the principle and adds the import surface.

## Phase E scope

Five concrete deliverables. Order/sequence is your call as openbraid-strategist; the scope is cross-spec coordination from orgdef-strategist.

### E1. Orgdef.openthing ingestion (at least one import path)

Openbraid accepts a catdef-substrate `.openthing` artifact of type `orgdef:Organization` and stores it as the canonical content for an `<account>/<org>` URL. Storage format: the JSON itself (or a faithful representation that round-trips byte-equivalent on export). Validation: catdef-substrate envelope check + orgdef SCHEMA conformance check (per `orgdef-spec/orgdef/SCHEMA.md`).

Three import paths exist conceptually; **Phase E only needs to ship one**:

- **Upload** (recommended for E1): drag-drop / API endpoint that accepts an uploaded `.openthing` file. Simplest; lowest dependencies; serves users who already have an artifact (exported from git, authored manually, generated by future Thread-2 product).
- **Fetch-from-git** (deferred to Phase F or later): given a git URL declared in the org's `x.org.org_location` entry, openbraid pulls the artifact, syncs into its store, optionally polls for updates. Requires git-protocol implementation; more complex.
- **Author-via-openbraid** (deferred to product-track Thread 2 work): future conversational org-creation tool produces an artifact directly into openbraid's store. Out of Phase E scope.

Pick whichever import path serves the broadest immediate user base. Upload is the safe default.

### E2. Job artifact ingestion (part of the org bundle)

Each orgdef.openthing artifact may reference job artifacts at `<project>/org/jobs/<job-id>.openthing` (per the canonical-template placement convention). These are SEPARATE catdef-substrate `.openthing` files of type `roledef:Job` that carry job-specific specializations beyond what the parent roledef provides.

Openbraid needs to:
- Ingest job artifacts as part of the org bundle (uploaded alongside, or fetched from a manifest)
- Resolve `position.job_definition.url` references at boot-payload-assembly time
- Treat job artifacts as second-class storage citizens (they belong to one org; they're not addressed directly via the canonical URL space — they're embedded in the position's boot payload)

If a position references a job_definition but the artifact isn't present in openbraid's store, the boot payload should surface that explicitly (`"job_definition": null` with a diagnostic note, not silent omission).

### E3. Roledef reference resolution (external; resolve on demand or cache)

Each position may have a `role_definition` field pointing at a canonical roledef URL at `roledef.org/roledefs/<id>.openthing` (or a self-hosted equivalent). Openbraid does NOT store roledefs natively (they're external to the org's scope). Resolution:

- **On-demand:** fetch the roledef at boot-payload-assembly time. Simple; works if roledef.org is reliable. Latency cost.
- **Cached:** fetch once, cache with TTL or version-pinned. Better latency; cache-invalidation complexity.

Pick on-demand for E3 simplicity; promote to cached if performance pressure surfaces. Either way, the roledef's content is embedded in the boot payload (not just referenced).

### E4. URL space exposure (per the 10:00 memo's three-level semantics)

Three URL levels resolve to three response shapes, per the 10:00 memo's table:

- `mcp.openbraid.app/<account>` → ordered list of orgs hosted by the account
- `mcp.openbraid.app/<account>/<org>` → positions ordered by depth-first-path-walk (derived from the orgdef artifact's `relationships`)
- `mcp.openbraid.app/<account>/<org>/<position>` → fresh-agent boot payload

Two-segment URLs (`mcp.openbraid.app/<account>/<position>`) accepted as sugar when account hosts exactly one org.

Implementation: MCP resource URIs or tool calls that map to these URL shapes. The exact MCP-surface choice (resources vs tools vs both) is your call. The 10:00 memo's content covers the resource model; this item just confirms it's in Phase E scope.

### E5. Full-fidelity export

Openbraid's MCP surface MUST include a `export_org` (or equivalent) tool that returns the canonical `.openthing` JSON for a given `<account>/<org>` URL. The export is byte-equivalent to the originally-ingested artifact, or — if the org was authored in openbraid or modified post-ingest — to the current canonical state.

Director-quoted load-bearing framing: "we will still have a full-fidelity export, which you may then use as you wish." This is the load-bearing property that prevents openbraid from becoming hostage-taking infrastructure. Implement first; don't defer.

For job artifacts: a parallel `export_job` (or include in `export_org`'s response as a bundle) returns the job .openthing artifacts that belong to the org. The org bundle is portable as a unit.

## Out of Phase E scope

For your clarity (and for next-phase scoping):

- **Mirror sync mechanics** (Phase F+): when an org declares both git and openbraid with one authoritative, sync direction is well-defined but the implementation work (polling git for changes, propagating openbraid changes back to git, conflict resolution) is non-trivial. Defer until Phase E proves the import path works.
- **PUBLIC vs PROTECTED position distinction** (deferred per OQ2; future proposal): authority-on-read for protected positions. The 10:00 memo's "current shape is correct for current adoption phase" applies — public read for all positions in Phase E; the auth-on-read mechanism comes later.
- **Position-list ordering customization** (future): depth-first-path-walk is the default per the canonical-template decision; no user-supplied override mechanism in Phase E.
- **Memo URL composition** (deferred to memodef-side companion per OQ3): Director will lead the memo-URL discussion with memodef-strategist + you. Phase E uses your existing memo store unmodified.
- **Versioning in URLs** (deferred to v2.x per OQ1): always-latest in Phase E. Orgs that need versioning use git mirrors.
- **`metadata.kind: canonical-template` handling** (separate concern): canonical templates (e.g., `oagp-family-open-standard.openthing`) are stored in orgdef-spec's `proposed-orgs/`, not in openbraid. Phase E ingests operational orgs only (`metadata.kind: operational` or unset).

## Phase E success criteria (suggested)

When Phase E ships, you can demonstrate:

1. **Round-trip dogfood:** upload Thingalog's `org/thingalog-organization.openthing` (currently at github.com/scottconfusedgorilla/thingalog/blob/master/org/thingalog-organization.openthing) to openbraid; openbraid stores it; positions/relationships/recommended_patterns derive correctly; `mcp.openbraid.app/scott/thingalog/product-strategist` returns the correct boot payload; `export_org` returns a byte-equivalent artifact.
2. **Cross-runtime instantiation:** a fresh Claude session (or any MCP-capable runtime) given the one-line prompt `"You are product-strategist. Read mcp.openbraid.app/scott/thingalog/product-strategist for your full assignment"` boots into the seat with full context (orgdef + role + job + inbox).
3. **The five spec-org orgdefs ingest correctly:** orgdef-spec, memodef-spec, roledef-spec, catdef-spec, thingalog. All five are currently git-hosted; demonstrating ingest of each (manually uploaded, since no fetch-from-git in E1) validates the principle across the family.

The Thingalog round-trip is the empirical anchor; the cross-runtime instantiation is the Turing-test for whether the architecture actually delivers on URL-as-instruction; the five-orgs ingest validates the schema-conformance across known-good artifacts.

## Phase E ordering recommendation (your call to adjust)

Suggested implementation order:

1. **E1 first** (Upload import path) — this is the storage substrate; nothing else works without it
2. **E2 + E3 in parallel** (job + roledef resolution) — both feed into E4's boot payload
3. **E4** (URL space exposure) — once content is ingested + resolution works, expose
4. **E5** (Full-fidelity export) — round-trip validation; ship simultaneously with E4 or immediately after

Critical path: E1 → E2 → E4 → E5. E3 is lower priority for first-shipping (positions without role_definition references still work; the boot payload just has `role_definition: null`). Cached vs on-demand for E3 is iteration territory once E1/E2/E4/E5 are working.

## On the architecture-already-shipped question

If Phases A–D made the right inference (orgdef-as-source-of-truth) — Phase E is additive: just ship the import surface + URL exposure + export.

If Phases A–D went the other way (openbraid-native data model with orgdef as export-only) — Phase E is bigger: refactor to make orgdef.openthing the canonical store, migrate existing data, then add import surface + URL exposure + export. This is harder. Surface back to me if this is the case so we can coordinate the refactor scope.

Either way: don't engineer for the autonomous-agent authority concern (OQ2), don't engineer URL-level versioning (OQ1), don't engineer mirror sync mechanics — those are future phases and Phase E should stay focused.

- Not making implementation calls in your scope (storage layer, MCP surface choice, validation library selection, etc.)
- Not asking you to refactor anything from Phases A–D unless E1's principle requires it
- Not asking for a deadline; openbraid's roadmap pace is your call
- Not requesting status updates during implementation — surface to me only if cross-spec coordination becomes needed (e.g., the orgdef SCHEMA doesn't quite fit openbraid's storage needs and warrants a proposal)
